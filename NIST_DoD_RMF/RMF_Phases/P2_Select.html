Phase 2: Select

Selection of Controls

  • Selection Objectives
  • Identify security controls needed
    • Tailoring Encompasses:
      • Identifying/designating common controls in initial baselines
      • Making risk-based decisions on remaining baseline controls
      • Selecting compensating controls
      • Supplementing baselines with additional controls and control enhancement, if applicable
  • Select minimum security control baseline
  • Build monitoring strategy for identified controls
  • Selection tasks
  • Common control identification
  • Security control selection

Selection Documents

  • FIPS-200: Minimum Security Requirements for Federal Information and Information Systems

    • OR

  • NIST SP 800-53r5: Security and Privacy Controls for Information Systems and Organizations

    • AND

  • Selection Input Documents:
    • FIPS-199 Document
    • System Security Plan (SSP) - so far
    • Common control inputs
    • Tailoring guidance
    • Initial Risk Assessment
  • Output = 3/4 SSP

There are 6 tasks in Security Selection:

  1. Control Selection
  2. Control Tailoring
  3. Control Allocation
  4. Documentation of planned control implementations
  5. Continuous monitoring strategy of the system
  6. Plan review and approval

The information system owner is the primary role in this function. The Common Control Provider, Security Architect, and Information Owner all play a supporting role.

Organization of Controls

  • Security controls are organized into eighteen families
    • Related by security function
    • Designated by a two-character identifier
    • Published in alphabetical order in the Appendix
  • Programmatic controls are listed in separate appendixes
  • Families of controls are organized into 3 classes:
    • Operational Controls - Day to day processes
    • Technical Controls - hardware and software
    • Maangement Controls - actions taken to manage the system

Common Controls

  • Provide consistent and more cost-effective security accross an organization
    • Accelerates implementation
    • Reduces cost
    • More consistent behavior
  • Common Control Provider (CCP)
    • Selected by CIO and CISO
    • Provides for inheritance of controls

      • may be external systems from the one being categorized

NIST SP 800-53 Implementation Details

Control selection baseline is the starting point established with the FIPS-199 and NIST 800-53 documents.

  • Tailoring Controls is accomplished through:
    • Scoping the system
    • Parameterization
    • Any compensating guidance provided
  • Supplementing through additional controls uses enhancements provided in the SP 800-53 families of controls
  • Additional consideration criteria are:
    • Operating environment
    • Organization-specific requirements
    • Threat assessments
    • Business Impact Analysis (BIA) for implemented controls

Control Categories

    Primary
  • Preventative
  • Detective
  • Corrective
    Secondary
  • Supplemental
  • Compensating
  • Deterrent

Kinds of Controls

  • Common
    • Controls that are inherited by one or more organizationa information systems - Like a common firewall
  • Hybrid
    • Controls that are implemented in part as Common and part System-specific
    • Like: MS AD Group Policy
  • System-specific controls
    • Control that is implemented entirely within the IS under review

Monitoring Strategy

  • Developed during the initial part of the System Development Life Cycle (SDLC)
  • Should allow for robust review and near real-time awareness of current security state of the system
  • Especially important in highly dynamic environments
  • There are 4 parts to Monitoring:
    1. Configuration Management
    2. Security impact analysis of any changes
    3. Continuous assessment of the system
    4. Security status reporting
  • Identifies which controls to be monitored, frequency, and assessment approach
  • Duration is determined by the trustworthiness of the common control provider

Review and Approval of the System Security Plan (SSP)

  • Review is conducted by the System Owner (SO) and IS Security Officer (ISSO)

    • Possibly requires CISO office review as well

  • Completed SSP is signed by the SO and the ISSO
  • SSP is presented to the Authorizing Official (AO) for approval
  • SSP is accepted and signed by the AO
    • Approves system boundaries
    • Approves (MSB) and defined risks
    • Accepts initial risks
