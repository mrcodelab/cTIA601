Phase 3: Implement

Implementation of Controls

  • Implementation objectives:
    • Install security controls into the system
  • The tailoring process involves:
    • Designating common controls
    • Applying scoping considerations
    • Selecting compensating controls
    • Supplementing the baseline controls
    • Providing implementation specifications
  • Implementation tasks
    • Security controls implementation
    • Security controls documentation
    • Primary Role: Information System Owner

Implementation Documents


NIST SP 800-53r5

  • Recommended security controls for federal Information systems
    • Provides guidelines for selecting and specifying security controls for the information system
    • Helps achieve more secure information systems by providing a "rich" catalog of security controls and measures
    • Also provides the foundation for assessing and measuring the effectiveness of the security controls and measures

      • Note: Revision 5 added the "IP" Individual Participation control group with 6 controls and the PA - Privacy Authorization which has 4 controls.

Tailoring of controls

    3 Primary Areas:
  • Scoping guidance
  • Compensating controls
    • selected from SP 800-53 Appendix F
    • Implemented when controls from a baseline cannot be implemented due to scoping considerations or other risk-based reasons
    • Provide protection comparable to the control that would have been implemented
    • i.e. a policy or procedure in lieu of an automated control
  • Organizational-defined parameter specifications
  • Scoping considerations help ensure that organizations select and implement only those controls needed for appropriate protection of the information systems
  • Apply scoping considerations as needed to assist with making risk-based decisions on security control selection
  • Scoping considerations are not intended to limit organizations in rendering risk-based decisions
    Supplementing the baseline
  • Sometimes the baseline controls are not sufficient to address specific threats and vulnerabilities
  • Inputs for supplementation may include risk assessment and/or regulations, policies, etc.
  • These are not the same as compensating controls
    • Compensating controls are put in place when a control cannot meet the current intent
    • Supplemental controls would be on top of compensating controls to address specific risks

Privacy - SP 800-53 Appendix J

  • Appendix J complements security controls in Appendix F
  • Based on the Fair Information Practices and Principles from the Privacy Act of 1974
  • Draws from the e-Governance Act of 2002, Section 208
  • Updates regularly with privacy-related OMB guidance
  • Note: Rev5 replaced "Chief Privacy Officer" with "Senior Agency Official for Privacy"

E-authentication Assurance Levels

  • Based on the OMB M04-04 and NIST SP 800-63-3
      • M04-04:
    • Level 1: Little or no confidence in the asserted identity's validity
    • Level 2: Some confidence in the asserted identity's validity
    • Level 3: High confidence in the asserted identity's validity
    • Level 4: Very high confidence in the asserted identity's validity
    Applying Controls - Documentation
  • Document the control implementations in Security System Plan (SSP)
    • Planned inputs
    • Expected behavior
    • Expected outputs
  • Functional description - the "As Implemented" details
  • Traceability matrix back to requirements
  • Platform dependencies
  • Responsible person/CCP

Types of Management Controls a.k.a Administrative Controls

    Documented policies, guidance and procedures:
  • Risk assessment and management policies/procedures
  • C&A process itself along with signed ATO's
  • Security planning for all systems
  • SDLC manuals and guidance
  • Security program management at higher organizational levels

Types of Physical Controls

  • Automated vs. manual fire detection, inspection, and suppression
  • Automated vs. manual temperature and humidity controls
  • Emergency lighting
  • Voltage regulation
  • Positioning of data displays
  • Data handling and dissemination
  • Facility guarding, access policies, and procedures

Types of Operational Controls

    Focus on the human element:
  • Policies
  • Procedures
  • Processes
  • Training & Awareness
    Documentation Includes:
  • Configuration management plan
  • Detailed IA architecture documentaiton
  • Hardware/software/firmware baseline inventories
  • Information assurance vulnerability management plan (IAVMP)
  • Incident repsonse plan
  • Disaster recovery & contingency plans
  • Memorandums of Agreement (MOA's)
  • Test plans, procedures, and results

Technical Controls - a.k.a. Logical Controls

Implemented using software, hardware or firmware that restricts logical access on an inforamtion technology system

Logs

    NIST SP 800-92: Guide to Computer Security Log Management, September 2006
  • Log sources
  • Analyze log data
    • Proactive log analysis shows potential problems and possible attack
    • Logs are also useful for establishing baselines, forensic analysis, and internal audits
  • Respond to identified events
  • Long term log data storage

Incident Response Handling

  • NIST SP 800-61r2: Computer Security Incident Handling Guide
  • Four Steps:
    1. Preparation
    2. Detection and analysis
    3. Containment, eradicaiton and recovery
    4. Post-incident activity

Incident Response Reporting

  • CAT 0 - Exercise/network defense testing
  • CAT 1 - *Unauthorized Access
  • CAT 2 - *Denial of Service (DoS)
  • CAT 3 - *Malicious code
  • CAT 4 - *Inappropriate usage
  • CAT 5 - Scans/probes/attempted access
  • CAT 6 - Investigation

    • * Any incident that involves compromised PII must be reported to US-CERT within 1 Hour of detection regardless of the incident category reporting timeframe

Incident Response Agencies:

  • US-CERT
  • DOD-CERT
  • Information Analysis Infrasturcture Protection (IAIP)
  • CERT Coordination Center (CERT/CC)
  • Information Sharing and Analysis Centers (ISAC)

Contingency Plan Criteria

  • Maximum Tolerable Downtime (MTD)
  • Recovery Time Objective (RTO) - how long you have to restore regular business operations
  • Recovery Point Objective (RPO) - normal operations
  • Work Recovery Time (WRT)
