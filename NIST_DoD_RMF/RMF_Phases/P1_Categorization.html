Phase 1: Categorization

  • Categorization of the information system is basesd on the impact analysis. It is performed to determine:
    • The types of information included within the security authorization boundary
    • The security requirements for the information types
    • The potential impact on the organization resulting from compromise
  • The result is used as the basis for developing the security plan, selecting security controls, and determining the risk inherent in operating the system

Task 1: Categorization Objective:

  • Produce the FIPS 199 Categorization document for the system
  • This document is utilized in multiple locations and on multiple efforts
    • SSP - Systems Security Plan
    • Budgets and CPIC activities
    • SORN's - System of Record Notifications
    • Program Objective Memoranda - POM and Program of Record - POR for service
  • Determination of the information types:
    • A specific categoy of information
    • Develop system descriptions
    • Register the system
  • Identify roles
    • Primary role = Information system owner
    • Support = Information owner and all others involved in risk assessments
FIPS 199

The "High Water Mark" is the same as Overall Residual Risk in Army Risk Assessments


Guidelines for adjusting system categorization
  • Aggregation - standard OPSEC aggregation concept
    • Combinations of data which increase CIA risks
    • "Total" > sum of parts
    • Especially common in PII & HIPPA areas
  • Critical system functionality
    • Impact on connected systems
    • Both connected to and receiving from systems - bi-directional risk
  • Extenuating circumstances
    • Catastrophic loss of system availability
    • Large Supporting Systems are involved
    • Interconnected systems
    • Public data requirements not already covered under FIPS 199
      • Public data requirements where Confidentiality = N/A
  • External factors
  • Public information integrity
  • critical Infrastructure and key resources
  • Trade secrets
  • Overall information system impact
  • Privacy Information

Task 2: Information Systems Description

Major Applications are made of Major Information Systems

    Under NIST SP 800-18: Guide for Developing Security Plans for Federal Information Systems
  • MA = Major Application
  • GSS = General Support System
    • Minor Applications - may inherit some or all security controls provided by the GSS or MA
    • Minor Apps must implement the minimum controls required for the information system if the edge controls don't do so
    Under OMB Circular A-130
  • AIS = Automated Information System
    • MA = Major Application
    • GSS = General Support System
  • Defines "Adequate security"
    Site Authorizations
  • Focus is one location or facility with all component parts of the system including inherited common controls from external providers
    • Usually under one command or facility
    • Common security policies
    • Common operating environment
  • Allows for single package for identical copies of system which are deployed in different environments
    Enclave Authorizations
  • A collection of computing environments under the control of a single authority and security policy
    • LAN
    • Backbone Networks
    • Data Centers
    • Ship - Naval IT
  • In the DoD, enclaves always assume the highest security level being supported
  • Enclave types
    • Core - Endpoints
    • Leverage - Where guest networks access from the internet, DMZs, DB's, Security Tools, Apps-hosting
    • Tenant - Projects hosting
    • Dev/Test Enclave is isolated Nobody talks to test
    • Core & Tenent talk through Leverage but never talk to each other directly
    Defining IS Boundaries
  • Defined by a set of resources allocated to the system
  • Support same mission/business objectives
  • Reside in the same operating environment

Interconnections

NIST SP 800-47
Is a security guide for interconnecting information systems. It sets the ground rules for interconnections.

  • The direct connection of 2 or more information systmes
  • Documented with a MOU/MOA which includes the T&C's and responsibilities for each organization
  • Is created at the current point in the system lifecycle when it is determined necessary
  • Not needed for endpoints
  • Assigns traceable responsibilities
  • Includes a topological diagram of the interconnections
